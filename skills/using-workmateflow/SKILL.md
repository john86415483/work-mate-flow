## 指令优先级

Skills 会覆盖默认的系统提示行为，但**用户的明确指令始终优先**：

1. **用户的明确指令**（CLAUDE.md、AGENTS.md、直接请求）— 最高优先级
2. **Skills** — 与默认系统行为冲突时覆盖之
3. **默认系统提示** — 最低优先级

## 如何访问 Skills

使用 `Skill` 工具。当你调用一个 skill 时，其内容会被加载并呈现给你 — 直接遵循它。切勿使用 Read 工具读取 skill 文件。

## 使用 Skills

### 规则

**在任何回复或行动之前调用相关或要求的 skill。** 即使某个 skill 只有 1% 的可能性适用，你也应该调用它来检查。如果调用的 skill 最终不适用于当前情况，你无需使用它。

### 红旗警示

以下想法意味着 STOP — 你在找借口：

| 想法 | 现实 |
|---------|---------|
| "这只是一个简单的问题" | 问题也是任务。检查是否有适用 skill。 |
| "我需要先了解更多上下文" | Skill 检查应该在澄清问题之前。 |
| "让我先探索一下代码库" | Skills 告诉你怎么探索。先检查。 |
| "我快速看下 git/文件就行" | 文件缺少对话上下文。检查是否有适用 skill。 |
| "让我先收集信息" | Skills 告诉你怎么收集信息。 |
| "这不需要正式的 skill" | 如果有 skill 存在，就用它。 |
| "我记得这个 skill" | Skills 会不断演变。读取当前版本。 |
| "这不算一个任务" | 行动 = 任务。检查是否有适用 skill。 |
| "用 skill 太小题大做了" | 简单的事情也会变得复杂。用它。 |
| "我先做这一件事" | 做任何事之前都要先检查。 |
| "这样做效率更高" | 无纪律的行动浪费更多时间。Skills 防止这种情况。 |
| "我知道那是什么意思" | 知道概念 ≠ 使用 skill。调用它。 |

### Skill 优先级

当多个 skill 可能适用时：

1. **流程型 skill 优先**（brainstorming、debugging）— 决定如何执行任务
2. **实现型 skill 第二** — 指导具体执行

### Skill 类型

**严格型**（TDD、debugging）：精确遵循。不要因省事而偏离。

**灵活型**（模式）：根据上下文调整原则。

Skill 本身会告诉你它属于哪种类型。

### 用户指令

指令说明 WHAT（做什么），而非 HOW（怎么做）。"添加 X"或"修复 Y"不意味着可以跳过工作流程。
